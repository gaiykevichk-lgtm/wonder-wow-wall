# Product Spec: Фото-редактор стен (Photo Wall Editor)

> **Статус:** draft
> **Домен (Bounded Context):** `visualizer`
> **Дата:** 2026-03-20
> **Приоритет:** Высокий (Этап 2 — Расширенный функционал)

---

## 1. Проблема и контекст

### Текущее состояние

Конструктор стен (`constructor` domain) работает на абстрактной сетке: пользователь задаёт размеры стены в ячейках, размещает панели snap-to-grid. Это полезно для планирования, но **не даёт представления**, как панели будут смотреться на реальной стене пользователя.

### Проблема

- Пользователь не может оценить, как дизайн впишется в его интерьер
- Невозможно учесть мебель, розетки, окна, двери — реальные препятствия на стене
- Конверсия страдает: пользователь не уверен в результате → откладывает покупку
- Конкуренты предлагают AR/фото-визуализацию

### Решение

**Движок фото-редактора стен** — пользователь загружает фото своей стены, система автоматически распознаёт поверхность стены и объекты (мебель, двери, окна, розетки), и позволяет размещать панели Wonder Wow Wall только на свободных участках стены с реалистичной визуализацией.

## 3. Пользовательский сценарий (User Flow)

### 3.1 Основной сценарий — Happy Path

```
1. Пользователь нажимает «Примерить на фото» (кнопка на странице каталога,
   товара или конструктора)
2. Загружает фото стены (drag & drop, выбор файла, или камера на мобильном)
3. Система анализирует фото:
   a. Определяет область стены (Wall Mask)
   b. Определяет объекты-препятствия (Object Mask): мебель, двери, окна,
      розетки, выключатели, картины, полки
   c. Определяет перспективу и геометрию стены
4. Пользователь видит фото с наложенной разметкой:
   - Зона стены подсвечена (можно размещать панели)
   - Объекты-препятствия обведены (нельзя размещать)
   - Пользователь может скорректировать маску вручную (добавить/убрать зоны)
5. Пользователь выбирает дизайн панелей из каталога
6. Выбирает размер панели (30×30, 30×60, 60×60 см)
7. Выбирает способ размещения:
   a. «Заполнить всю стену» — автозаполнение свободной поверхности
   b. «Разместить вручную» — клик по зонам стены
   c. «Акцентная зона» — выделить область для панелей
8. Система накладывает панели на фото с учётом:
   - Перспективы (трансформация по геометрии стены)
   - Освещения (анализ теней/света на фото)
   - Обхода объектов-препятствий
9. Пользователь видит реалистичный предпросмотр
10. Может менять дизайн, цвет, размер — предпросмотр обновляется в реальном времени
11. Видит расчёт стоимости (количество панелей × цена)
12. Нажимает «Добавить в корзину» или «Сохранить проект»
```

### 3.2 Альтернативные сценарии

| Сценарий | Поведение |
|----------|----------|
| Фото без стены (крупный план мебели) | Предупреждение: «Стена не найдена. Попробуйте другое фото» |
| Стена занята >80% объектами | Предупреждение: «На стене мало свободного места. Показать доступные зоны?» |
| Низкое качество / размытое фото | Предупреждение: «Качество фото недостаточно для анализа. Загрузите фото с лучшим разрешением» |
| Несколько стен на фото (угол комнаты) | Система предлагает выбрать стену для редактирования |
| Пользователь хочет покрыть область за объектом | Ручная коррекция маски: «Игнорировать объект» |

## 5. Нефункциональные требования

| Категория | Требование |
|-----------|-----------|
| **Производительность** | Сегментация: ≤ 5 сек (GPU), ≤ 10 сек (CPU). Рендеринг: ≤ 200 мс |
| **Точность** | IoU ≥ 0.85 для wall segmentation на интерьерных фото |
| **Форматы** | JPEG, PNG, HEIC, WebP. Вход: до 20 MB. Рабочее: до 2048px |
| **Адаптивность** | Desktop (1920+), Tablet (768+), Mobile (375+). Touch на мобильном |
| **Приватность** | Фото обрабатываются на сервере, хранятся не более 24 ч после анализа, не передаются третьим лицам. Соответствие 152-ФЗ |
| **Офлайн** | Fallback: ручная отрисовка маски, если сервер недоступен |
| **A11y** | Keyboard navigation для инструментов, alt-текст, высокий контраст для масок |

## 6. DDD-архитектура (Bounded Context: `visualizer`)

### 6.1 Почему отдельный Bounded Context

Фото-редактор стен — это отдельный ограниченный контекст `visualizer`, а не расширение `constructor`, потому что:

1. **Другая модель данных**: работа с изображениями, масками, перспективой vs абстрактная сетка
2. **Другой Ubiquitous Language**: сегментация, маска, калибровка, перспектива — терминология computer vision
3. **Другие зависимости**: ML-модели, обработка изображений, GPU-ресурсы
4. **Независимый жизненный цикл**: может развиваться отдельно от grid-конструктора

### 6.2 Ubiquitous Language — Единый язык домена `visualizer`

| Термин (RU) | Термин (EN) | Определение |
|-------------|-------------|-------------|
| Сцена | Scene | Загруженное фото интерьера с метаданными анализа |
| Маска стены | WallMask | Бинарная карта: какие пиксели являются стеной |
| Маска объектов | ObjectMask | Карта распознанных объектов с классами (мебель, двери, окна и др.) |
| Сегментация | Segmentation | Процесс разделения фото на стену и объекты |
| Калибровка | Calibration | Определение реального масштаба по фото |
| Перспектива | Perspective | Геометрия стены на фото (трапеция → прямоугольник) |
| Раскладка | Layout | Результат размещения панелей на стене |
| Акцентная зона | AccentZone | Выделенная пользователем область стены для панелей |
| Визуализация | Visualization | Реалистичный рендер панелей поверх фото |
| Проект визуализации | VisualizationProject | Сохранённая сцена + раскладка + настройки |

### 6.6 Infrastructure Layer

| Компонент | Реализация |
|-----------|-----------|
| `SegmentationService` | Python backend: модель сегментации (DeepLabV3+ / Segment Anything / OneFormer) |
| `PerspectiveDetector` | OpenCV: поиск линий, vanishing points |
| `VisualizationRenderer` | Frontend: HTML5 Canvas / WebGL + шейдеры перспективы |
| `PanelLayoutEngine` | Frontend: bin-packing алгоритм с учётом маски |
| `ScaleCalibrator` | Frontend: расчёт по 2 точкам; Backend: авто-детекция по размерам объектов |
| `ProjectRepository` | Backend: PostgreSQL + S3 (хранение фото и масок) |

### 6.7 Связи с другими Bounded Contexts

```
visualizer ──uses──→ catalog (дизайны панелей: id, name, image, price)
visualizer ──uses──→ order (добавление раскладки в корзину: CartItem[])
visualizer ──uses──→ subscription (проверка статуса: накладки бесплатно?)
visualizer ──uses──→ shared/config (размеры панелей, цены)
```

**Принцип**: Visualizer импортирует только **типы** из соседних доменов. Общение — через Anti-Corruption Layer (ACL): `visualizer` мапит свои `PlacedPanel` → `CartItem` через адаптер.

| POST | `/api/visualizer/projects/{id}/export` | Экспорт визуализации в изображение |

## 8. Технические подходы к сегментации

### Вариант A: Серверная ML-модель (рекомендуемый)

| Параметр | Значение |
|----------|---------|
| **Модель** | Segment Anything Model 2 (SAM 2) от Meta или OneFormer |
| **Фреймворк** | PyTorch + ONNX Runtime |
| **GPU** | NVIDIA T4 / A10 (inference) |
| **Latency** | ~2–5 сек (SAM 2), ~3–8 сек (OneFormer) |
| **Точность** | IoU ≥ 0.90 для стен |

**Плюсы**: высокая точность, поддержка сложных сцен, fine-tuning на интерьерных фото.
**Минусы**: требует GPU-сервер, сетевая задержка.

### Вариант B: Клиентская модель (TensorFlow.js)

| Параметр | Значение |
|----------|---------|
| **Модель** | DeepLabV3+ (MobileNetV2 backbone) |
| **Фреймворк** | TensorFlow.js + WebGL backend |
| **Latency** | ~5–15 сек (зависит от устройства) |
| **Точность** | IoU ~0.75–0.80 |

**Плюсы**: нет серверных затрат, работает офлайн.
**Минусы**: ниже точность, медленно на слабых устройствах, ограничены по памяти.

### Вариант C: Гибридный подход (рекомендуемый для MVP)

1. **Backend**: SAM 2 для точной сегментации (primary)
2. **Frontend**: Лёгкая ONNX-модель для быстрого preview (fallback)
3. Пользователь видит грубый preview через 2–3 сек (клиент), точный результат через 5–7 сек (сервер)

## 9. Алгоритм размещения панелей (Layout Engine)

### 9.1 Вход

- `WallMask`: бинарная маска стены (2D array)
- `ObjectMask`: маска объектов с классами
- `PerspectiveTransform`: матрица перспективы
- `PanelSize`: выбранный размер (30×30, 30×60, 60×60)
- `PlacementMode`: auto / manual / accent

### 9.2 Алгоритм авторазмещения (режим «Заполнить стену»)

```
1. Привести WallMask к прямоугольной проекции (убрать перспективу)
2. Наложить сетку панелей (шаг = размер панели в пикселях с учётом масштаба)
3. Для каждой ячейки сетки:
   a. Проверить, что ≥ 70% ячейки попадает в WallMask (порог заполнения)
   b. Проверить, что ≤ 10% ячейки пересекается с ObjectMask
   c. Если условия выполнены → разместить панель
   d. Иначе → пропустить ячейку
4. Применить обратную перспективу к координатам панелей
5. Результат: список PlacedPanel[] с координатами на фото
```

### 9.3 Алгоритм акцентной зоны

```
1. Пользователь рисует прямоугольник на фото (AccentZone)
2. Пересечь AccentZone с WallMask → рабочая область
3. Вычесть ObjectMask из рабочей области
4. Применить алгоритм авторазмещения внутри рабочей области
```

## 11. Edge Cases и ограничения

| Кейс | Решение |
|------|---------|
| Стена с обоями/текстурой | Сегментация должна распознавать текстурированные стены |
| Стена из кирпича / камня | Распознаётся как стена; предупреждение о поверхности |
| Зеркало на стене | Распознаётся как объект (декор), маскируется |
| Стена в тени | Адаптация яркости панелей к уровню освещения зоны |
| Панорамное фото | Поддержка фото с высоким aspect ratio (> 2:1) |
| Фото с людьми | Люди распознаются как объекты, маскируются |
| Очень маленькая свободная зона | Минимум: 1 панель 30×30 должна помещаться |

## 12. Метрики успеха

| Метрика | Цель | Как измеряем |
|---------|------|-------------|
| Конверсия фото → корзина | ≥ 25% | Загрузили фото → добавили проект в корзину |
| Средний чек (фото-проекты) | +30% vs обычный конструктор | Средний чек заказов из фото-редактора |
| Точность сегментации | IoU ≥ 0.85 | Тесты на валидационном датасете интерьерных фото |
| Ручная коррекция маски | ≤ 40% пользователей корректируют | % сессий, где использовалась кисть/ластик |
| Время до предпросмотра | ≤ 10 сек | Медиана: загрузка фото → первый рендер панелей |
| NPS фото-редактора | ≥ 50 | Опрос после использования |

## 13. Зависимости и риски

| Риск | Вероятность | Влияние | Митигация |
|------|------------|---------|-----------|
| Низкая точность сегментации на нестандартных фото | Средняя | Высокое | Ручная коррекция маски + fine-tuning модели на интерьерных фото |
| Медленный inference без GPU | Высокая | Среднее | Гибридный подход (клиент preview + сервер точный) |
| Большой размер ML-модели (клиент) | Средняя | Среднее | Lazy-загрузка модели, серверный fallback |
| Перспективная трансформация неточна | Средняя | Среднее | Ручная корректировка 4 точек перспективы |
| Высокие затраты на GPU-сервер | Средняя | Среднее | Rate limiting, очередь задач, auto-scaling |

## 15. Acceptance Criteria (критерии приёмки MVP)

- [ ] Пользователь может загрузить фото стены (JPEG/PNG, до 20 MB)
- [ ] Система определяет область стены и объекты за ≤ 10 сек
- [ ] Маска стены отображается поверх фото (зелёный) и объекты (красный)
- [ ] Пользователь может корректировать маску кистью и ластиком
- [ ] Пользователь может выбрать дизайн, размер и цвет панелей
- [ ] Панели автоматически размещаются на свободной поверхности стены
- [ ] Панели НЕ накладываются на распознанные объекты
- [ ] Отображается количество панелей, площадь и стоимость
- [ ] Пользователь может добавить проект в корзину
- [ ] Фото-редактор доступен по маршруту `/visualizer`
- [ ] Интерфейс соответствует дизайн-системе (DESIGN-SYSTEM.md)
